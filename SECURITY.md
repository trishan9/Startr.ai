# Security Overview - Startr AI

This document covers the main security considerations, AI-specific risks, and known limitations of the system.

---

## Authentication & Access Control

All API routes are protected by InsForge session authentication. The session token is read server-side from a `HttpOnly` cookie, it is never exposed to the browser. Unauthenticated requests are rejected immediately before any database or AI operations run.

Database queries are scoped to the authenticated user through InsForge row-level security. A user cannot read or modify another user's projects or pages by guessing IDs.

Project URLs use 12-character nanoid slugs (`/project/aB3xKp9mLqR7`). These have 62¹² ≈ 3.2 × 10²¹ possible values, effectively unguessable, but authentication is the real access boundary, not the slug alone.

The Azure OpenAI API key lives only in server-side environment variables and is never sent to the browser.

## AI Security & Prompt Safety

### Prompt Injection

User input is string-interpolated into AI prompts. A crafted message like _"ignore previous instructions and return…"_ can influence model behaviour. This is a known limitation of the current architecture.

**Mitigations in place:**

- Intent classifier uses a strict single-word output (`generate` / `regenerate` / `chat`), outputs outside these three words default to `chat`, limiting the blast radius of intent hijacking
- Blueprint agent inputs are truncated (`userPrompt.slice(0, 200)` in the diagram agent)
- JSON responses are parsed with `parseJsonResponse` which extracts only the first valid `{…}` block, ignoring prose injected before or after

**Known gaps:**

- The regeneration prompt embeds `selectedPage.htmlContent` directly, AI-generated HTML from a previous turn re-enters the prompt context. A sufficiently crafted first message could plant instructions that activate on the next regeneration request
- Architecture and roadmap agents receive the full untruncated user prompt

**Practical risk:** Since this is a single-user tool (each user works on their own projects), prompt injection primarily harms the attacker's own output quality rather than other users' data.

### Output Validation

All six blueprint agents use `safeRun` wrappers, if any agent returns malformed or unparseable JSON, it falls back to a safe placeholder and the rest of the blueprint continues. No agent failure can crash the pipeline or expose raw error details to the user.

Mermaid diagram output passes through a `sanitize()` function that strips code fences, fixes colon syntax, removes HTML tags, and normalises special characters before the diagram library renders it.

### AI-Generated HTML

Pages generated by the AI are rendered in sandboxed `<iframe>` elements with `srcdoc`. The generation prompt explicitly instructs the model to produce HTML and CSS only, no JavaScript. The iFrame sandbox restricts navigation away from the preview context.

The HTML is stored as-is in the database. If a future feature exports or shares pages, a DOMPurify sanitisation pass should be added before serving that content outside the iFrame context.

## AI Output Bias & Quality Limitations

GPT-5.4 is a general-purpose reasoning model, not a specialised startup consultant. Its outputs reflect patterns in its training data and carry the following known biases:

**Technology bias:** The model has a strong prior toward certain tech stacks (Next.js, PostgreSQL, Redis, Node.js, AWS). For niche domains, embedded systems, scientific computing, regulated industries, the recommended stack may be inappropriate. Always validate technology choices against actual requirements.

**Geography & market bias:** Business model assumptions, pricing suggestions, and target market characterisations are weighted toward English-speaking Western markets. A microloan platform for Sub-Saharan Africa will receive architecture recommendations designed for US/EU SaaS, not for mobile-money infrastructure or low-bandwidth environments.

**Scale bias:** The model defaults to mid-scale SaaS architecture regardless of the actual product scale. A simple internal tool does not need Redis, a CDN, and microservices, but the architecture agent may recommend them anyway.

**Recency cutoff:** The model's training has a knowledge cutoff. Very recent frameworks, APIs, or regulatory requirements (e.g., new data protection laws, new cloud services) will not appear in recommendations.

**The blueprint is a starting point, not a specification.** It should be reviewed by a domain expert before any engineering or business decisions are made based on it.

## Rate Limiting & Cost Protection

The Azure OpenAI deployment is capped at 10 requests per minute. The system enforces this with a software rate limiter (6.5 s minimum gap between calls) and an HTTP 429 retry handler. This prevents the blueprint pipeline from exceeding the quota limit.

There is currently no per-user daily request limit. In production, a daily cap per account (e.g., 10 full generations per day) should be enforced at the database level to prevent unintentional cost exhaustion.

## Data Handling

**What is stored:**

- User account data managed by InsForge Auth
- Project metadata (title, slugId, timestamps)
- Generated HTML pages (can be 50–200 KB each)
- Chat message history including the full blueprint JSON

**What is not stored:**

- Passwords (managed entirely by InsForge Auth, not the application)
- Payment information (none collected)
- The Azure API key (environment variable only, excluded from version control via `.gitignore`)

**Conversation context:** The last 10+ messages are sent from the browser to the server on each request and used as AI context. This means the full conversation (including any sensitive information a user typed) travels over the network on every turn. HTTPS encrypts this in transit.

## Known Limitations to Address Before Production

| Area             | Issue                                       | Recommended Fix                                         |
| ---------------- | ------------------------------------------- | ------------------------------------------------------- |
| Rate limiting    | No per-user request cap                     | Add daily quota enforced in DB                          |
| Prompt injection | Raw user text in agent prompts              | Wrap in XML delimiters, add input length cap            |
| Message history  | Full history sent from browser              | Reload from DB server-side instead                      |
| CSP headers      | None configured                             | Add `Content-Security-Policy` in `next.config.ts`       |
| Input validation | `slugId` and `selectedPageId` not validated | Regex check before DB queries                           |
| Logging          | `console.log(error)` may leak stack traces  | Replace with structured logger, redact sensitive fields |
